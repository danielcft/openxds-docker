There are two types of configuration files for OpenXDS. project level configuration and actor level configuration. The project level configuration files define general project related properties, while the actor level configuration files mainly define actor connection information.

Project Configuration

openxds.properties- the main project level properties file which contains the main properties

applicationContext.xml- the spring context file. Usually it is not modified unless more control is needed

omar.properties- the properties for the freebxml omar configuration

repository.jdbc.cfg.xml- jdbc connection for the omar repository. (As the omar repository is not used by OpenXDS, it should be removed in the future)

Actor Configuration

OpenXDS have sample actor configuration files located at the conf/actors folder. These files were successfully used for mesa tests and Connectathon tests. You are free to copy the whole actors folder and create your own configuration files. The actors folder has a bunch of xml configuration files. The starting xml file is IheActors.xml; all the other xml files can be linked from the IheActors.xml. This file contains connections, actors, or files that describe actors or connections. The certs folder under actors contains private key and public certificate used by secure transactions.

Understand Configuration Files

Note: The xml syntax including tags, attributes and elements is all case-insensitive.

Tags

Configuration Tag

Configuration
 Each XML configuration file has a root Configuration tag.

 

ConnectionFile Tag:

ConnectionFile
 Specifies a file that defines standard connection and/or secure connection configuration.
Attributes
File
 Required
 The file name of this ConnectionFile.

 

Ex: <ConnectionFile file="AuditRepositoryConnections.xml" />

Actor Tag:

Actor
 Defines an actor such as XDS Repository, XDS Registry, Xca Initiating Gateway or XCA Responding Gateway.
Attributes
Name
 Required
 The name of this actor. Any name can be given, but be sure to avoid duplication.
Type
 Required
 Use XdsRepository for an XDS Repository actor,
XdsRegistry for an XDS Registry actor,
XcaIG for an XCA Initiating Gateway actor,
XcaRG for an XCA Responding Gateway actor, and
SecureNode for an Audit Trail client actor.
Elements and Attributes
Description
 Optional
 The detailed description of this actor
Server
 Required
 Describes the server connection of this actor.
      - connection
 Required
 The connection name of this server connection.
Note: all connections including the ones described below have to be defined in one of the Connections.xml files
PixServer
 Required for XdsRegistry
 Describes a PIX Registry Server that accepts PIX Feed messages from a PIX Source or PIX Manager. It is applicable to the XdsRegistry actor.
      - connection
 Required
 The connection name of this PIX registry server, defined in XdsRegistryConnections.xml
Registry
 Required for XdsRepostiory and XcaRG,
Optional for XcaIG
 Describes a client connection for this actor to connect to a Document Registry actor. It is applicable to the XdsRepository, XcaRG and XcaIG actors.
      - connection
 Required
 The connection name for the client side of the Registry
Repository
 Required for XcaRG,
Optional for XcaIG
 Describes a client connection for this actor to connect to a Document Repository actor. It is applicable to the XcaRG and XcaIG actors.
    -connection
 Required
 The connection name for the client side of the Repository
Responding Gateways
 Required for XcaIG Describes a list of Responding Gateways that this Initiating Gateway actor connects to. It is applicable to only the XcaIG actor.
Responding Gateway
 At one is required in RespondingGateways Describes a Responding Gateway connection that this Initiating Gateway actor can connects to. It is applicable to the XcaRG and XcaIG actors.
     - homeId
 Required in RespondingGateway
 The home community id of this responding gateway
     - query
 Required in RespondingGateway
 The connection name for the cross gateway query transaction of this responding gateway
     - retrieve
 Required in RespondingGateway
 The connection name for the cross gateway retrieve transaction of this responding gateway
AuditTrail
 Required for SecureNode
 Describes an audit trail type. Applicable to all actors.
    -consumer
 Required in AuditTrail
 The connection to an audit record repository consuming the audit messages.

 


IheActors.xml Example:

<Configuration>

    <!--Defines the configuration of Audit Repository Server for Audit Client-->

    <ConnectionFile file="AuditRepositoryConnections.xml" />

    <!--Defines the configuration of XDS Registry -->

    <ConnectionFile file="XdsRegistryConnections.xml" />

    <!--Defines the configuration of XDS Repository-->

    <ConnectionFile file="XdsRepositoryConnections.xml" />

    <!--Defines the configuration of XCA Responding Gateway-->

    <ConnectionFile file="XcaRespondingGatewayConnections.xml" />

    <!--Defines the configuration of XCA Initiating Gateway-->

    <ConnectionFile file="XcaInitiatingGatewayConnections.xml" />

    <!--The XDS Registry actor configuration-->

    <Actor name="xdsreg" type="XdsRegistry">

      <Description>XDS Registry</Description>

      <Server connection="xds-registry" />

      <PixServer connection="pix-registry" />

    </Actor>

    <Actor name="xdsreg_secure" type="XdsRegistry">

      <Description>Secure XDS Registry </Description>

      <Server connection="secure-xds-registry" />

      <PixServer connection="secure-pix-registry" />

    </Actor>

    <!--The XDS Repository actor configuration-->

    <Actor name="xdsrep" type="XdsRepository">

      <Description>XDS Repository</Description>

      <Server connection="xds-repository" />

      <Registry connection="xds-registry-client" />

    </Actor>

    <Actor name="xdsrep_secure" type="XdsRegistry">

      <Description>Secure XDS Repository</Description>

      <Server connection="secure-xds-repository" />

      <Registry connection="secure-xds-registry-client" />

    </Actor>

    <!--The XCA Responding Gateway actor configuration-->

    <Actor name="xcarg" type="XcaRG">

      <Description>XCA Responding Gateway</Description>

      <Server connection="xca-rg" />

      <Registry connection="misys-registry" />

      <Repository connection="misys-repository" />

    </Actor>

    <Actor name="xcasrg" type="XcaRG">

      <Description>Secure XCA Responding Gateway</Description>

      <Server connection="xca-secure-rg" />

      <Registry connection="misys-registry" />

      <Repository connection="misys-repository" />

    <!--The XCA Initiating Gateway actor configuration-->

    <Actor name="xcaig" type="XcaIG">

      <Description>XCA Initiating Gateway</Description>

      <Server connection="xca-ig" />

      <Registry connection="local-registry" />

      <Repository connection="local-repository" />

      <RespondingGateways>

        <RespondingGateway homeId="urn:oid:1.3.6.1.4.1.21367.2010.1.2.2050" query="ibm-rg-query" retrieve="ibm-rg-retrieve"/>

        <RespondingGateway homeId="urn:oid:1.3.6.1.4.1.21367.2010.1.2.2025" query="vangent-rg" retrieve="vangent-rg"/>

        <RespondingGateway homeId="urn:oid:1.3.6.1.4.1.21367.2010.1.2.2010" query="spirit-rg" retrieve="spirit-rg"/>

        <RespondingGateway homeId="urn:oid:1.3.6.1.4.1.21367.2010.1.2.2040" query="hxti-rg" retrieve="hxti-rg"/>

      </RespondingGateways>

    </Actor>

    <Actor name="xcasig" type="XcaIG">

      <Description>Secure XCA Initiating Gateway</Description>

      <Server connection="xca-secure-ig" />

      <Registry connection="local-secure-registry" />

      <Repository connection="local-secure-repository" />

      <RespondingGateways>

        <RespondingGateway homeId="urn:oid:1.3.6.1.4.1.21367.2010.1.2.2050" query="ibm-secure-rg-query" retrieve="ibm-secure-rg-retrieve"/>

        <RespondingGateway homeId="urn:oid:1.3.6.1.4.1.21367.2010.1.2.2025" query="vangent-secure-rg" retrieve="vangent-secure-rg"/>

        <RespondingGateway homeId="urn:oid:1.3.6.1.4.1.21367.2010.1.2.2010" query="spirit-rg" retrieve="spirit-rg"/>

        <RespondingGateway homeId="urn:oid:1.3.6.1.4.1.21367.2010.1.2.2040" query="hxti-rg" retrieve="hxti-rg"/>

</Configuration>    

 

StandardConnection Tag:


StandardConnection
 Defines a non-TLS connection.
Attributes
Name
 Required
 The name of this connection, which must be referenced by the Actors defined in IheActor.xml.
Elements
HostName Optional for server; required for Client.
 The host name or ip address that this connection connects to.
Use localhost if this connection is for a server socket.
Port
 Required
 The port of this connection
Url
 Required for the client connetion
 The url part of this connection
Identifier
 -NamespaceId
  -UniversalId
  -UniversalIdType
 		Required
 The identifier tag specifies an assigning authority, or an identity. It can include 3 elements: NamespaceId, UniversalId and UniversalIdType. There are two ways to specify an Identity: one way is to use NamespceId; the other is to use the combination of UniversalId and UniversalIdType. If both NamespaceId and the combination of UniversalId and UniversalIdType are used, they must be consistent and refer to the same entity.
Property
 Required and Optional
 The Property tag defines a property for this connection.
There are multiple properties defined for each actor connection. Some are required; some others optional.
PropertySet
 Optional
 Defines a group of related properties.
   -Entry
 Optional
 Each entry defines a specific property for this PropertySet.
IncludeFile
 Optional
 This is an inline element. It can be used to place a trunk of properties in a separate xml file. The main purpose of the IncludeFile is to reuse some common configuration.


 

Example for XdsRegistry connection:

    <StandardConnection name="xds-registry">

        <!--The host name of the Registry Server-->

        <HostName>localhost</HostName>

        <!--The port -->

        <Port>8010</Port>

        <Property name="sourceIds" value="1.3.6.1.4.1.21367.2010.1.2" />

        <Includefile name="XdsCodes.xml" />

        <Identifier type="domain" name="GLOBAL">

            <NamespaceId>IHENA</NamespaceId>

            <UniversalId>1.3.6.1.4.1.21367.2010.1.2.300</UniversalId>

            <UniversalIdType>ISO</UniversalIdType>

        </Identifier>

    </StandardConnection>

 

SecureConnection Tag:

SecureConnection

 Defines a TLS connection.
It has the same attributes and elements as the corresponding StandardConnection. In addition, a SecureConnection must include the following additional Elements:

KeyStore

 Required
 The file containing the Key Store for this connection. By default, it is in the certs folder.
KeyPass
 Required
The password for the Key Store.
TrustStore
 Required
The file containing the Trust Store for this connection. By default, it is in the certs folder.
TrustPass
 Required
The password for the Trust Store.

 

Example for Trust and KeyStore in SecureConnection tag.

<!-- Trust and Keystore -->

 <!-- Used by Connectathon 2009 -->

 <KeyStore>certs/OpenPIXPDQ_2009_KEY.p12</KeyStore>

 <KeyPass>password</KeyPass>

 <TrustStore>certs/OpenPIXPDQ_2009_STORE.jks</TrustStore>

 <TrustPass>password</TrustPass>
